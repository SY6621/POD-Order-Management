## **可行性结论与更好的建议**
- **可行性**：Web + IDE 做一个“自然语言→Skill Spec→校验→dry-run”的小工具完全可行，且与你的项目（订单解析、效果图生成、标签生成、日志审计）天然匹配。
- **你想法的强点**：你已经有清晰模块化架构与开发规范，这让 Skill 的“规则校验器”更容易落地（例如统一异常、日志、测试）。
- **需要补齐的点（建议你直接纳入 MVP）**  
  - **状态机**：订单从 `received → parsed → mockup_sent → revision_requested → approved → production_ready → shipped` 之类的迁移必须在 Skill 层声明，否则人工介入时最容易乱。  
  - **幂等键**：邮件/客户回复重复触发是常态，Skill 必须声明幂等键（例如 `email_message_id + order_id + mockup_revision`）。  
  - **工件版本化**：效果图/标签是工件（artifact），必须有 `artifact_id`/`revision`，否则你无法稳定“重复发送/回滚/对账”。  
  - **人工介入协议**：不是“需要人工”，而是“谁介入、介入要填什么、介入后如何恢复自动化”。  
  - **可回归测试**：每个 Skill 至少 1 正例 + 1 反例（解析失败、邮箱认证失败、客户回复不完整等），否则生成出来无法在开发期稳定复用。

## **6 个 Skills 模版（按你的项目链路与例子定制）**
下面每个 Skill 都用同一套结构：**目的 / 触发 / 输入输出 / 副作用与权限 / 执行计划 / dry-run 校验 / 最小测试**。你后续做工具时就用这套当“规范输出格式”。

### 1) IngestEtsyOrderEmail（邮件读取与过滤）
- **目的**：从 IMAP 读取 Etsy 订单相关邮件，筛选出待处理消息并产生标准“邮件事件”
- **触发**：定时轮询；或手动（IDE 一键）
- **输入**：邮箱配置引用（非明文）、筛选条件（发件人/主题关键字/时间窗）
- **输出**：`email_event[]`（含 message_id、received_at、raw_content_ref、thread_key）
- **副作用/权限**：IMAP 只读；写日志；可写 DB（仅记录 message_id 去重）
- **执行计划**：连接→拉取→过滤→去重→产出事件
- **dry-run 校验**：secrets key 是否齐；过滤条件是否过宽；是否声明幂等键（message_id）
- **最小测试**：一封符合规则邮件应产出 1 个事件；重复运行不应产出重复事件

### 2) ParseOrderFromEmail（非结构化→标准订单模型）
- **目的**：把邮件文本解析为标准订单模型（order_id、customer、items、total、address 等）
- **触发**：收到 `email_event`
- **输入**：`email_event`、解析策略（订单格式版本/模板）
- **输出**：`order_draft`（含字段完整性评分、缺失字段列表、解析置信度）
- **副作用/权限**：无外部写操作；仅日志
- **执行计划**：格式识别→字段抽取→规范化→校验（必填字段）
- **dry-run 校验**：是否声明“解析失败分类”；是否产出缺失字段清单（用于人工介入或二次提问）
- **最小测试**：标准邮件解析成功；异常邮件必须输出可定位的失败原因与缺失字段

### 3) UpsertOrderAndIndexArtifacts（落库 + 幂等 + 工件索引）
- **目的**：把 `order_draft` 写入数据库，保证幂等；建立工件索引（后续 mockup/label 版本归档）
- **触发**：收到 `order_draft`
- **输入**：`order_draft`、幂等键策略、状态机起始状态
- **输出**：`order_record`（含 order_id、db_id、current_status、correlation_id）
- **副作用/权限**：DB 读写；日志；可选文件系统索引目录初始化
- **执行计划**：查重→插入/更新→写审计→设置状态
- **dry-run 校验**：声明事务边界；写操作是否有审计；是否声明 allowed 状态迁移
- **最小测试**：同一 order_id 重复 upsert 不应产生重复记录；状态迁移非法必须失败

### 4) GenerateEffectMockup（效果图生成）
- **目的**：根据订单商品信息生成效果图工件（支持 revision）
- **触发**：订单进入 `parsed/ready_for_mockup`
- **输入**：`order_record`、模板/素材引用、输出规格（尺寸/格式）
- **输出**：`artifact`（artifact_id、type=mockup、revision、file_ref、checksum）
- **副作用/权限**：文件系统写；可选 DB 写（artifact 索引）；日志
- **执行计划**：加载素材→渲染→导出→校验（尺寸/透明通道等）→登记工件
- **dry-run 校验**：输出规格是否完整；文件落盘路径策略是否声明；是否声明清理策略
- **最小测试**：给定固定输入能生成工件；缺素材/渲染失败能产出可定位错误

### 5) CustomerMockupApprovalLoop（发送确认 + 客户修改 + 人工介入 + 再发送）
- **目的**：把效果图+确认邮件发送给客户；接收客户修改请求；触发人工介入；生成新 revision 并重复发送；最终进入确认状态
- **触发**：mockup artifact 生成完成；或收到客户回复邮件/后台“客户要求修改”事件
- **输入**：`order_record`、`artifact(mockup)`、客户联系信息、沟通模板引用、人工介入规则
- **输出**：`approval_state`（pending/changes_requested/approved）、`next_action`
- **副作用/权限**：邮件发送；DB 写状态；审计；可创建“人工任务”（task/ticket）
- **执行计划**：发送→等待/轮询回复→分类（同意/修改/疑问）→若修改则创建人工任务→人工完成后触发生成新 revision→重新发送→直至 approved
- **dry-run 校验**：必须声明 reviewer_role、resume_strategy；必须声明“如何识别同一线程”（thread_key/correlation_id）；必须声明最大轮次与超时策略
- **最小测试**：同意回复应推进到 approved；修改请求应进入 changes_requested 并生成人工任务；重复回复不应导致状态混乱

### 6) GenerateShippingLabelAfterApproval（确认后进入生产/物流标签）
- **目的**：订单 approved 后生成物流标签（PDF）并登记工件，进入生产/发货环节
- **触发**：订单状态 `approved → production_ready`
- **输入**：`order_record`、收货地址、物流参数（承运商/服务级别）
- **输出**：`artifact`（type=shipping_label、revision、file_ref）
- **副作用/权限**：文件系统写；可选调用物流 API；DB 写状态与审计
- **执行计划**：校验地址→生成/调用 API→导出 PDF→登记工件→更新状态
- **dry-run 校验**：地址字段完整性；外部 API 超时/重试；状态迁移合法性
- **最小测试**：缺地址必须失败并给缺失字段；成功生成应落库并产生 artifact 记录

## **最高效使用这些 Skills 的路径（开发期最省沟通成本）**
- **先固化“状态机 + 工件模型 + 幂等键”三个公共约定**：这决定你后续所有 Skill 的稳定性与可验证性，避免每次重新解释。
- **写功能先写 Skill，不先写代码**：每个新功能先用自然语言生成/更新 Skill Spec，然后跑校验与 dry-run；通过后再进入编码。
- **把 Skill 当作仓库资产管理**：每次需求变更只改 Skill Spec（版本递增），IDE 自动生成 diff/测试用例；你写代码时只引用 Skill 名称与契约。
- **人工介入统一走“任务化”**：所有人工介入都必须落到同一种结构（who/what/inputs/decision/resume），否则系统越做越乱。
- **建立最小回归集**：把常见邮件样本、修改请求样本、失败样本变成 test fixtures；你每次改解析/模板都能立刻 dry-run 回归。
