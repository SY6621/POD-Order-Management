> **版本号格式**：POD V{大版本}.{小版本}_{日期}  
> **示例**：POD V1.05_20260327

## 1. 版本号规则

### 1.1 格式定义

```
POD V{大版本}.{小版本}_{YYYYMMDD}
```

### 1.2 版本号说明

| 类型 | 规则 | 示例 |
|------|------|------|
| **大版本** | 功能重大升级时递增 | V2.0、V3.0、V4.0 |
| **小版本** | 功能完成/迭代时递增 | V1.00 → V1.01 → V1.02 |
| **日期** | 提交当天的日期 | 20260327 |

## 2. 备份时机

### 2.1 必须备份的情况

1. **每日开始工作前** - 检查并拉取最新代码
2. **完成一个功能模块后** - 如客服外链、设计器等
3. **进行重大修改前** - 作为安全 checkpoint
4. **下班前** - 确保当日工作已保存

### 2.2 备份检查清单

```
□ 功能开发完成
□ 代码已测试通过
□ 文档已更新（如需要）
□ 版本号已更新
□ 提交信息完整
```

## 3. 备份操作流程

### 3.1 每日开始

```powershell
# 1. 进入项目目录
cd d:\ETSY_Order_Automation

# 2. 检查当前状态
git status

# 3. 拉取最新代码
git pull origin main
```

### 3.2 完成功能后备份

```powershell
# 1. 添加所有修改
git add -A

# 2. 提交（使用正确的版本号格式）
git commit -m "POD V1.06_20260328: 功能名称 - 简要说明"

# 3. 推送到GitHub
git push origin main
```

### 3.3 强制推送（仅在修改提交信息时使用）

```powershell
git push origin main --force
```

> ⚠️ **警告**：强制推送会覆盖远程历史，仅在版本号修正等必要时使用

## 5. 分支管理

### 5.1 主分支

- **main** - 生产环境代码，始终保持可发布状态

### 5.2 开发分支（可选）

```powershell
# 创建功能分支
git checkout -b feature/生产中订单页面

# 切换回主分支
git checkout main

# 合并功能分支
git merge feature/生产中订单页面
```

## 6. 常见问题

### 6.1 冲突处理

```powershell
# 1. 先拉取最新代码
git pull origin main

# 2. 手动解决冲突

# 3. 添加解决后的文件
git add .

# 4. 继续提交
git commit -m "解决冲突: xxx"
```

### 6.2 回退操作

```powershell
# 查看提交历史
git log --oneline

# 回退到上一个提交（保留修改）
git reset --soft HEAD~1

# 回退到指定提交（完全回退）
git reset --hard {commit_hash}
```
